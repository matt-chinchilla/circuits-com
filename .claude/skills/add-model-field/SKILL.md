# add-model-field

## When to run

Any user prompt matching patterns like:

- "Add a `tax_id` string field to Supplier"
- "Add a `release_year` integer to Part"
- "Add a nullable `parent_company` VARCHAR to Supplier"
- "Suppliers need an `is_verified` boolean"

## Inputs (collect via AskUserQuestion if not already in the prompt)

| Arg | Example | Notes |
|---|---|---|
| `model_py` | `Supplier` | PascalCase — the Python class name |
| `model_snake` | `supplier` | lowercase singular |
| `model_plural` | `suppliers` | lowercase plural (for routes path) |
| `field_name` | `tax_id` | snake_case |
| `sql_type` | `String(32)` \| `Integer` \| `Boolean` \| `UUID(as_uuid=True)` \| `DateTime(timezone=True)` | SQLAlchemy type |
| `ts_type` | `string` \| `number` \| `boolean` \| `string \| null` | TypeScript type |
| `nullable` | `True` / `False` | |
| `default` | `None`, `False`, literal, or `lambda: datetime.now(timezone.utc)` | Python default expression |
| `admin_only` | `True` / `False` | If True, field goes ONLY on admin endpoints — NOT public `<Model>Response` |

## Files to edit — exactly 6 places (per CLAUDE.md "Adding a Supplier field requires 6 files")

### 1. `api/app/models/<model_snake>.py`

Insert into the model class:

```python
<field_name> = Column(<sql_type>, nullable=<nullable>, default=<default>)
```

Place it adjacent to other similar-type columns for readability. Do NOT reorder existing columns.

### 2. Alembic migration (autogenerated, then reviewed)

```bash
cd api && alembic revision --autogenerate -m "add <field_name> to <model_snake>"
```

Then open the generated file at `api/alembic/versions/<new>.py` and review it. `--autogenerate` catches column additions but misses:
- Data backfills (if nullable=False and default is computed from other columns)
- Index additions that aren't declared on the model
- CHECK constraints

If the field is nullable=False with no default, ADD a backfill step before the `add_column` and an `alter_column` step after to flip nullable to False. SQLite (test DB) doesn't enforce NOT NULL additions mid-migration the same way Postgres does — watch prod.

### 3. `api/app/schemas/<model_snake>.py` (or wherever Pydantic schemas live)

Add the field to:
- **`<Model>Response`** — ONLY if `admin_only=False`. CRITICAL: `SupplierResponse` is shared between public `/api/suppliers/` and admin `/api/admin/suppliers/` — adding admin-only fields here exposes them publicly.
- **`<Model>Create`** — always, unless field is autogenerated (created_at-style)
- **`<Model>Update`** — always, Optional-wrapped so partial updates work

Pydantic fragment:

```python
<field_name>: <python_type> | None = None  # or concrete type if non-nullable
```

If `admin_only=True`, create or extend `Admin<Model>Response` alongside and wire routes to use it when the caller is authenticated as admin.

### 4. `api/app/routes/<model_plural>.py` → `<model_snake>_to_dict` helper

Add one line:

```python
"<field_name>": obj.<field_name>,
```

Place next to similar-type attributes for readability.

### 5. `frontend/src/types/admin.ts`

Add to the `Admin<Model>` interface:

```ts
<field_name>: <ts_type>;
```

### 6. `frontend/src/pages/admin/<Model>sPage.tsx` + `<Model>FormPage.tsx`

Two sub-edits:

**In `<Model>sPage.tsx`** — add the field to `COLUMNS` so it appears in the admin list view. Decide if it's visible-by-default or behind a "show more" toggle based on typical admin UX (wide columns should be opt-in).

**In `<Model>FormPage.tsx`** — three places in one file:
- Form field in the `<form>` (input/select matching `ts_type`)
- Payload construction in the submit handler (`body.<field_name> = formState.<field_name>`)
- Review-section output (so the "Review & confirm" step shows the new field to the admin)

## Critical gotcha — re-enforced

From CLAUDE.md: `SupplierResponse` is shared by public AND admin endpoints — fields added there appear in the unauthenticated `/api/suppliers/`. If the field is sensitive (contact email, internal notes, revenue, etc.), use `admin_only=True`.

Also: tests use SQLite via `Base.metadata.create_all` — schema generated from the SQLAlchemy model, NOT from Alembic. This means tests pass as soon as you edit the model; the migration is only exercised against prod Postgres. ALWAYS test the migration locally via `alembic upgrade head` + `alembic downgrade -1` before merging.

## Output contract

Produce a single summary message BEFORE touching files, listing:

1. Every file to edit with its absolute path
2. The exact snippet to add to each
3. The `alembic revision` command
4. The expected test/verify commands:
   - `cd api && alembic upgrade head`
   - `cd api && alembic downgrade -1 && alembic upgrade head` (roundtrip test)
   - `cd api && python -m pytest tests/ -q`
   - Frontend auto-rebuilds via the `frontend-rebuild` hook
5. Manual verify: visit `http://localhost/admin/<model_plural>` and create/edit one

Then wait for the user to confirm the plan before making the edits.

## Rollback plan (include in the summary)

```bash
cd api && alembic downgrade -1
git checkout -- api/app/models/<model_snake>.py \
                api/app/schemas/<model_snake>.py \
                api/app/routes/<model_plural>.py \
                frontend/src/types/admin.ts
# Manual undo for <Model>sPage.tsx + <Model>FormPage.tsx edits via git checkout or diff
```

## Verification after edits

1. Run `cd api && alembic upgrade head` — must exit 0
2. Run `cd api && python -m pytest tests/ -q` — must exit 0 (the PostToolUse pytest hook will fire automatically after the .py edits too)
3. Frontend auto-rebuild hook fires on TS edits; wait ~20s, then hard-refresh `http://localhost/admin/<model_plural>`
4. Create one record with the new field populated, save, and confirm it persists across reload
5. Downgrade test: `cd api && alembic downgrade -1` — the field disappears cleanly, `alembic upgrade head` restores it
